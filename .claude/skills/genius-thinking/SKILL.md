# Genius Thinking Framework v2

## Overview

이 프레임워크는 **탐색, 재정의, 우선순위화, 아이디어 품질**이 중요한 분석/설계/기획 작업에서 사용합니다.

---

## 운영 원칙

1. 사고 공식은 **과학적 수식**이 아니라 **은유적 체크리스트**이다
2. 점수/계산은 **내부 정렬/우선순위화 용도**로만 쓰고, 결과는 **근거 중심 서술**로 출력
3. 정보가 부족하면:
   - (a) 필요한 정보 목록
   - (b) 현재 가능한 가정(Assumptions) 목록
   - (c) 가정이 결과에 미치는 영향(리스크)
   을 먼저 제시한 뒤 진행
4. 한 번에 10개 다 쓰지 말고 **2~4개만 선택**

---

## 출력 포맷

### A) 문제 정의 (1문장)
한 문장으로 "지금 해결할 문제"를 정의

### B) 제약/목표/가정
- 제약 (비용/시간/정책/기술/인력 등)
- 목표 (성공 기준, KPI)
- 가정 (불확실한 부분을 어떻게 가정했는지)

### C) 선택한 사고 공식(2~4개) + 선택 이유
아래 10개 중 2~4개를 고르고, 왜 골랐는지 설명

### D) 심층 분석
- 관찰(Observation)
- 패턴/원인(Pattern & Cause)
- 레버리지 포인트(Leverage)
- 리스크/반례(Risk & Counterexample)
- 결론(So what)

### E) 아이디어 (10개 이상)
각 아이디어:
- 아이디어명
- 핵심 메커니즘
- 기대효과
- 실행 가능성(F)
- 위험(R) + 리스크 유형
- 최소 실험(1~2주 검증 방법)

### F) Top 3 추천안 + 이유

### G) 1~2주 진화(TE) 루프 계획
- 학습(L) / 경험(E) / 반성(R) 포함

## 사고 공식 세트

### (1) GI - 천재적 통찰 도출 공식
```
GI = (O × C × P × S) / (A + B + ε)
```
- O: 관찰의 깊이 (1–10)
- C: 연결의 독창성 (1–10)
- P: 패턴 인식 능력 (1–10)
- S: 종합적 사고 (1–10)
- A: 고정관념 수준 (1–10)
- B: 편향 정도 (1–10)

### (2) MDA - 다차원적 분석 프레임워크
```
MDA = Σ[Di × Wi × Ii]
```
- Di: i번째 차원의 통찰 점수
- Wi: 우선순위 가중치 (0.5~2.0)
- Ii: 파급 영향력 (1.0~3.0)

**권장 차원:**
- D1: 시간 (과거–현재–미래)
- D2: 공간 (로컬–글로벌)
- D3: 추상 (구체–추상)
- D4: 인과 (원인–과정–결과)
- D5: 계층 (미시–거시)

### (3) CC - 창의적 연결 매트릭스
```
CC = w1·|A ∩ B| + w2·|A ⊕ B| + w3·f(A→B)
```
- |A ∩ B|: 공통점 개수
- |A ⊕ B|: 차이점 개수
- f(A→B): A→B 전이 가능 정도 (0~10)
- 기본 가중치: w1=1, w2=2, w3=2

### (4) PR - 문제 재정의 알고리즘
```
PR = M(ψ) ∘ S(φ) ∘ T(θ) (P₀)
```
- P₀: 원래 문제
- T(θ): 관점 회전 (사용자/운영자/재무/보안/법무/고객)
- S(φ): 범위 확장/축소
- M(ψ): 메타레벨 이동 (기능→원칙→시스템→인센티브)

### (5) IS - 혁신적 솔루션 생성 공식
```
IS = Σ[(Ci × Ni × Fi × Vi) / (Ri + ε)]
```
- Ci: 조합 방식 (1–10)
- Ni: 참신성 (1–10)
- Fi: 실행 가능성 (1–10)
- Vi: 가치 창출도 (1–10)
- Ri: 위험도 (1–10)

**실무 게이트:**
- 법/규제 리스크 임계치 초과 → 보류
- 보안 리스크 임계치 초과 → 대체 설계 후 재평가
- 운영 리스크 임계치 초과 → 단계적 롤아웃/가드레일

**최종 단계 — Verifiable Success Criterion 변환:**

> 출처: Andrej Karpathy의 LLM 효율성 관찰 (2026년 트윗) — Karpathy 4원칙 P4 (Goal-Driven Execution)
> LLM은 명확한 success criteria가 있을 때 가장 효과적으로 동작함. 캐노니컬 정의는 아래 "부록: LLM 코딩 4원칙" 참조

선정된 솔루션을 실행 단계로 넘기기 전, **명령형 지시를 검증 가능한 기준으로 재정의**:

| 명령형 (모호) | 검증 가능 (구체) |
|------------|---------------|
| "검증 추가해" | 잘못된 입력 케이스 3개 정의, 각각 테스트 작성 후 통과시키기 |
| "에러 처리 개선" | 재현 시나리오 명시, 각 시나리오의 예상 동작 사전 합의 |
| "성능 최적화" | 벤치마크 작성, 목표 수치(예: P95 < 200ms) 합의 후 측정 |
| "리팩토링" | 리팩토링 전후 동일 입력에 동일 출력 보장하는 회귀 테스트 |
| "문서 보강" | 독자/용도 정의, 검증 질문 3개에 답할 수 있는지 확인 |
| "보안 강화" | 위협 모델 정의, 각 위협에 대한 통제 수단 명시 |

**원칙**: 사용자/상위 단계에서 명령형으로 지시받았다면, 실행에 들어가기 전 **무엇으로 완료를 측정할지** 명시적으로 정의하고 합의한다.

**자기 점검:**
- [ ] 작업 완료 신호가 측정 가능한가?
- [ ] "다 됐다"고 말할 수 있는 객관적 기준이 있는가?
- [ ] 기준이 모호하면 한 단계 더 구체화 — 그래도 모호하면 사용자에게 질문

### (6) IA - 인사이트 증폭 공식
```
IA = I₀ × (1 + r)ⁿ × C × Q
```
- I₀: 초기 인사이트
- r: 개선율 (0~0.3)
- n: 반복 횟수
- C: 협력 효과 (1~3)
- Q: 질문의 질 (1~5)

### (7) TE - 사고의 진화 방정식
```
TE(t) = T₀ + ∫₀ᵗ [L(τ) + E(τ) + R(τ)] dτ
```
- T₀: 초기 사고 상태
- L: 학습 (문헌/데이터/사례)
- E: 경험 (실험/프로토타입)
- R: 반성 (회고/원인분석)

### (8) CS - 복잡성 해결 매트릭스
```
CS = (Nodes + Edges + Cycles + CouplingScore) × InteractionRisk
```
- Nodes: 구성요소 수
- Edges: 의존/연결 수
- Cycles: 순환 의존 수
- CouplingScore: 결합도 (1~10)
- InteractionRisk: 상호작용 리스크 (1~3)

### (9) IL - 직관적 도약 공식
```
IL = (log(1 + S) × E × T) / ((L + ε) × (R + ε))
```
- S: 정적 사고 시간
- E: 경험 축적
- T: 직관 신뢰
- L: 논리 제약
- R: 과도한 합리화

### (10) IW - 통합적 지혜 공식
```
IW = (K + U + W + C + A) × H × E
```
- K: 지식, U: 이해, W: 지혜, C: 공감, A: 실행력
- H: 겸손, E: 윤리
- H나 E가 낮으면 IW 급락

## 용도별 권장 조합

| 용도 | 권장 조합 | 이유 |
|------|----------|------|
| **제품/서비스 기획** | MDA + PR + IS | 문제 정의 → 다각도 분석 → 솔루션 우선순위 |
| **시스템 설계/아키텍처** | GI + MDA + CS | 복잡한 시스템을 부분/관계/레버리지로 분해 |
| **전략 수립 (OKR, 정책)** | PR + IS + TE | 시간에 따른 학습과 조정 |
| **창의적 아이데이션** | CC + PR + IL | 도메인 전이/메타포/직관 점프 |
| **Agent/Template 설계** | PR + MDA + IS | 역할 정의 → 다각도 분석 → 기능 우선순위 |

---

## 실행 절차

1. 문제(P₀)를 1문장으로 정의
2. 공식 2~4개 선택 + 선택 이유
3. PR로 문제 재정의
4. MDA로 다각도 분석
5. CC로 교차 도메인 연결 (필요시)
6. IS로 아이디어 평가/우선순위화 (게이트 포함)
7. Top 3 추천 + TE(1~2주 루프)로 마무리

## 부록: LLM 코딩 4원칙 (Karpathy Principles)

> 출처: Andrej Karpathy의 LLM 페일모드 관찰 (2026년 트윗)
> 이 4원칙은 코드 작성 단계에서 LLM의 전형적 실수를 차단하기 위한 가드레일이며,
> myclaude의 여러 자산(`clean-code`, `@Impl`, `@CTO`)에서 참조한다.

### 캐노니컬 ID 정의

| ID | 원칙명 (영문) | 원칙명 (국문) | 핵심 |
|----|------------|------------|------|
| **P1** | Think Before Coding | 가정 점검 | 검증 안 된 가정으로 진행 금지 |
| **P2** | Simplicity First | 과설계 방지 | 100줄로 풀 것을 1000줄로 풀지 마라 |
| **P3** | Surgical Changes | 정밀 편집 | 사용자 요청 범위 외 변경 금지 |
| **P4** | Goal-Driven Execution | 검증 가능한 성공 기준 | 명령형 → 측정 가능한 기준으로 변환 |

### myclaude 내 적용 위치

| 원칙 | 정의 위치 (single source) | 적용 시점 (참조) |
|------|----------------------|--------------|
| **P1** | `@Impl` Pre-flight Checklist | 코드 작성 전 |
| **P2** | `@Impl` Pre-flight + `@CTO` Failure Mode | 작성 전 + 리뷰 |
| **P3** | `clean-code` "Surgical Edit Discipline" | clean-code 참조 모든 agent |
| **P4** | `genius-thinking` IS 공식 — Verifiable Success Criterion (위 섹션) | 솔루션 평가 단계 + @Impl + @CTO |

### 사용 가이드

- 코드 작성 agent (`@Impl` 등)는 **P1-P4 모두** Pre-flight 자문
- 코드 리뷰 agent (`@CTO`)는 **P1-P4 모두** Failure Mode 점검
- 분석/설계 agent (`@Architect`, `@Product` 등)는 **P4 (검증 가능한 성공 기준)** 적용 — IS 변환 표 활용
- 4원칙 참조 시 위 ID(P1-P4) 사용 권장 — 용어 일관성 확보
